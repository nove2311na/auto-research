## Feature 1: Multi-Project Client-First Library System

### Problem

Every Webflow project has different CF class values (colors, spacing, typography tokens).
Previously there was no way to store, look up, or sync a per-project class library.
The global `knowledge-base/client-first-class-map.json` covers CF methodology but not
project-specific token values.

### Solution

Per-project library directories keyed by Webflow site ID, with a registry index and
Figma → library → Webflow sync pipeline.

## Feature 2: Parallel-Section Webflow Build Workflow

### Problem

Phase 2 (Webflow build) was single-operator and serial. For pages with many sections,
this is slow. Parallel subagent builds are blocked by a class-naming race: two agents
could invent the same class name with different values.

### Solution

HTML-contract-driven parallel build with strict phase split and apply-only subagents:

1. **Parent agent** reads Figma → emits ONE HTML contract (all sections, final CF class names).
   This serial authorship is the only place class names are decided.
2. **Phase 2A (serial, parent):** create new classes on Webflow + register into per-project library;
   create N section container elements in order → capture `parent_node_id` per section.
3. **Phase 2B (parallel):** spawn N `section-builder` subagents, each scoped to one `parent_node_id`.
   Subagents are apply-only — they recreate the HTML contract via native ops, never create classes.

HTML is an intermediate build contract, never a push payload (`whtml_builder` stays forbidden).

### Race-Prevention Logic (validate_build_contract.py)

```python
new_classes: dict[str, str] = {}  # name -> serialized definition
for entry in data.get("new_classes", []):
    definition = json.dumps({k: entry.get(k) for k in REQUIRED_NEW_CLASS_FIELDS}, sort_keys=True)
    if cls_name in new_classes and new_classes[cls_name] != definition:
        failures.append(f"class '{cls_name}' declared with conflicting definitions (naming-race risk)")
    new_classes[cls_name] = definition
```

| `UnicodeEncodeError` in `update_library_from_figma.py` on Windows cp1252 | Replaced `→` with `->` and `—` with `--` in all print statements. |
